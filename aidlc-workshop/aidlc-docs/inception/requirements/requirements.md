# 테이블오더 서비스 - 요구사항 정의서

## Intent Analysis Summary

| 항목 | 내용 |
|---|---|
| **User Request** | 테이블오더 서비스 (디지털 주문 플랫폼) 신규 구축 |
| **Request Type** | New Project (Greenfield) |
| **Scope Estimate** | System-wide (고객 UI + 관리자 UI + API 서버 + 데이터베이스) |
| **Complexity Estimate** | Complex (실시간 SSE, JWT 인증, 멀티테넌시, 세션 관리) |
| **Requirements Depth** | Comprehensive |

---

## 1. 프로젝트 개요

### 1.1 서비스 비전
디지털 주문 시스템을 통해 고객에게는 편리한 주문 경험을, 매장 운영자에게는 효율적인 운영 환경을 제공하는 테이블오더 플랫폼.

### 1.2 핵심 가치 제안
- **고객**: 대기 시간 없는 즉시 주문, 직관적 인터페이스
- **매장 운영자**: 인건비 절감, 실시간 주문 관리 및 모니터링
- **시장 차별화**: 다양한 업종 적용 가능, 직관적 관리자 도구

---

## 2. 기술 스택 결정사항

| 영역 | 기술 | 비고 |
|---|---|---|
| **Backend** | Python + FastAPI | 비동기 지원, SSE 네이티브 지원 |
| **Frontend** | React (TypeScript) | SPA, 컴포넌트 기반 UI |
| **Database** | PostgreSQL | 관계형 DB, 멀티테넌시 지원 |
| **Real-time** | Server-Sent Events (SSE) | 관리자 + 고객 양쪽 적용 |
| **Auth** | JWT (관리자) | bcrypt 해싱, 고객은 인증 없음 (URL 기반 접근) |
| **Deployment** | 로컬 서버 (On-premises) | 개발 및 운영 환경 |
| **Image Storage** | 로컬 파일 시스템 | 서버 직접 업로드 |

## 3. 기능 요구사항 (Functional Requirements)

### 3.1 고객용 기능 (Customer Features)

#### FR-C01: 테이블 식별 (URL 기반 접근)
- 고객은 별도 로그인/인증 없이 URL로 직접 접근
- URL에 매장 ID + 테이블 번호 포함 (예: `/store/{store_id}/table/{table_number}`)
- 관리자가 태블릿에 해당 URL을 1회 설정
- 유효하지 않은 매장/테이블 조합 시 에러 페이지 표시

#### FR-C02: 메뉴 조회 및 탐색
- 메뉴 화면이 기본(홈) 화면
- 카테고리별 메뉴 분류 및 표시
- 메뉴 상세 정보: 메뉴명, 가격, 설명, 이미지
- 카테고리 간 빠른 이동 (탭/사이드바)
- 카드 형태 레이아웃, 터치 친화적 (최소 44x44px)

#### FR-C03: 장바구니 관리
- 메뉴 추가/삭제, 수량 조절 (증가/감소)
- 총 금액 실시간 계산
- 장바구니 비우기
- 클라이언트 측 localStorage 저장 (페이지 새로고침 시 유지)
- 서버 전송은 주문 확정 시에만 수행

#### FR-C04: 주문 생성
- 주문 내역 최종 확인 화면
- 주문 확정 버튼
- 주문 성공 시: 주문 번호 표시 → 장바구니 자동 비우기 → 5초 후 메뉴 화면 자동 리다이렉트
- 주문 실패 시: 에러 메시지 표시, 장바구니 유지
- 주문 정보: 매장 ID, 테이블 ID, 메뉴 목록(메뉴명, 수량, 단가), 총 금액, 세션 ID

#### FR-C05: 주문 내역 조회
- 현재 테이블 세션 주문만 표시 (이전 세션 제외)
- 주문 시간 순 정렬
- 주문별 상세: 주문 번호, 시각, 메뉴/수량, 금액, 상태
- **주문 상태 실시간 업데이트 (SSE 적용)**
- 페이지네이션 또는 무한 스크롤

### 3.2 관리자용 기능 (Admin Features)

#### FR-A01: 매장 인증
- 매장 식별자 + 사용자명 + 비밀번호 입력
- JWT 토큰 기반 인증, 16시간 세션 유지
- 브라우저 새로고침 시 세션 유지
- 16시간 후 자동 로그아웃
- 비밀번호 bcrypt 해싱
- 로그인 시도 제한

#### FR-A02: 실시간 주문 모니터링
- SSE 기반 실시간 주문 업데이트 (2초 이내)
- 그리드/대시보드 레이아웃: 테이블별 카드 형태
- 각 테이블 카드: 총 주문액, 최신 주문 미리보기
- 주문 카드 클릭 시 전체 메뉴 목록 상세 보기
- **주문 상태 변경: 대기중 → 접수 → 준비중 → 완료 (4단계)**
- 신규 주문 시각적 강조 (색상 변경, 애니메이션)
- 테이블별 필터링 기능

#### FR-A03: 테이블 관리
- **테이블 초기 설정**: 테이블 번호 등록 및 매장 매핑 (PIN 없음), 접속 URL 생성
- **주문 삭제**: 대기중/접수 상태만 삭제 가능, 준비중/완료 상태는 삭제 불가 → 확인 팝업 → 즉시 삭제 → 총 주문액 재계산 → 피드백
- **테이블 세션 종료 (이용 완료)**: 확인 팝업 → 주문 내역 과거 이력 이동 → 현재 주문/총액 리셋 → 피드백
- **과거 주문 내역 조회**: 테이블별 과거 주문 목록 (시간 역순), 날짜 필터링

#### FR-A04: 메뉴 관리
- 메뉴 CRUD (등록/조회/수정/삭제)
- 메뉴 정보: 메뉴명, 가격, 설명, 카테고리, 이미지 (파일 업로드)
- 카테고리별 조회
- 메뉴 노출 순서 조정
- 필수 필드 검증, 가격 범위 검증

## 4. 비기능 요구사항 (Non-Functional Requirements)

### NFR-01: 성능
- SSE 주문 알림 2초 이내 전달
- 동시 접속: 매장당 테이블 10~30개, 동시 주문 10~20건 처리
- API 응답 시간: 일반 요청 500ms 이내

### NFR-02: 사용성 (Usability)
- **고객 UI**: 7~10인치 태블릿 중심 + 모바일 지원 반응형
- **관리자 UI**: 데스크톱 + 태블릿 + 모바일 모든 디바이스 반응형
- 터치 친화적 (최소 44x44px 터치 타겟)
- 카드 기반 레이아웃, 명확한 시각적 계층 구조

### NFR-03: 신뢰성
- 주문 데이터 무결성 보장 (트랜잭션 처리)
- SSE 연결 끊김 시 자동 재연결
- 에러 발생 시 사용자 친화적 메시지 표시

### NFR-04: 보안
- JWT 토큰 기반 관리자 인증 (16시간 만료)
- bcrypt 비밀번호 해싱
- 로그인 시도 제한
- 고객 접근: 인증 없음 (URL 기반 매장/테이블 식별)

### NFR-05: 확장성
- 멀티테넌시: 하나의 서버에서 여러 매장 독립 데이터 관리
- 매장별 데이터 격리

### NFR-06: 배포
- 로컬 서버 (On-premises) 배포
- 개발 환경 로컬 실행 지원

---

## 5. 멀티테넌시 구조

- 하나의 서버 인스턴스에서 다중 매장 지원
- 각 매장은 독립적인 데이터 (메뉴, 테이블, 주문)
- 매장 식별자(store_id)를 통한 데이터 격리
- 모든 API 요청에 매장 컨텍스트 포함

---

## 6. 주문 상태 흐름

```
대기중(PENDING) → 접수(ACCEPTED) → 준비중(PREPARING) → 완료(COMPLETED)
```

- **대기중**: 고객이 주문 생성 직후
- **접수**: 관리자가 주문 확인/접수
- **준비중**: 음식 준비 시작
- **완료**: 음식 준비 완료/서빙 완료

---

## 7. 테이블 세션 라이프사이클

```
[테이블 비어있음] → 고객 첫 주문 → [세션 시작 (세션 ID 자동 생성)]
                                        ↓
                                  추가 주문 → 동일 세션 ID에 귀속
                                        ↓
                              관리자 "이용 완료" → [세션 종료]
                                        ↓
                              주문 내역 → 과거 이력 이동
                              현재 주문/총액 → 리셋
                                        ↓
                              [테이블 비어있음] (새 고객 대기)
                                        ↓
                              다음 고객 첫 주문 → [새 세션 시작]
```

**핵심 규칙**:
- 세션은 테이블의 첫 주문 시 자동 생성된다
- 관리자가 "이용 완료" 처리하기 전까지 모든 추가 주문은 동일 세션에 귀속된다
- 세션 종료 시에만 주문 내역이 과거 이력으로 이동하고 테이블이 리셋된다

## 8. 이미지 관리

- 메뉴 이미지: 서버 로컬 파일 시스템에 직접 업로드
- 이미지 리사이징/최적화는 범위 외 (원본 그대로 저장 및 서빙)
- 업로드 API 제공, 메뉴 등록/수정 시 이미지 파일 첨부

## 9. 제외 기능 (Out of Scope)

| 카테고리 | 제외 항목 |
|---|---|
| **결제** | 실제 결제 처리, PG사 연동, 영수증, 환불, 포인트/쿠폰 |

| **고급 기능** | 데이터 분석, 매출 리포트, 직원 관리, 예약, 리뷰, 다국어 |
| **외부 연동** | 배달 플랫폼, POS, 소셜 미디어, 지도 API, 번역 API |

## 10. 용어 정의

| 용어 | 정의 |
|---|---|
| **테이블 세션** | 고객의 첫 주문부터 관리자의 "이용 완료" 처리까지의 기간 |
| **자동 로그인** | 1회 설정 후 저장된 정보로 자동 인증 |
| **SSE** | Server-Sent Events, 서버→클라이언트 단방향 실시간 통신 |
| **JWT** | JSON Web Token, 관리자 인증용 토큰 |
| **멀티테넌시** | 하나의 서버에서 여러 매장의 독립 데이터 관리 |
