# Unit of Work Plan - 테이블오더 서비스

## 목적
시스템을 관리 가능한 개발 단위(Unit of Work)로 분해하고, 각 유닛의 책임 범위, 의존성, 스토리 매핑을 정의합니다.

---

## 분해 질문

### Q1. 배포 모델
시스템을 어떤 배포 단위로 구성할까요?

- **A**: 모놀리스 (Backend 1개 + Frontend 2개 = 3개 배포 단위) — 단순, MVP에 적합
- **B**: 마이크로서비스 (도메인별 Backend 분리 + Frontend 2개) — 독립 배포, 복잡도 증가

[Answer]: B

### Q2. 개발 순서 전략
유닛 간 개발 순서를 어떻게 정할까요?

- **A**: 백엔드 우선 (API 완성 → 프론트엔드 개발) — API 안정성 확보 후 UI 개발
- **B**: 기능 단위 수직 슬라이스 (백엔드+프론트엔드 동시 개발) — 기능별 완성, 빠른 피드백
- **C**: 데이터 모델 우선 (DB 스키마 → API → UI) — 데이터 구조 확정 후 순차 개발

[Answer]: C

### Q3. 백엔드 모듈 분리 기준
모놀리스 백엔드 내부의 모듈을 어떻게 분리할까요?

- **A**: 도메인 기준 (주문, 메뉴, 테이블, 인증, SSE) — 비즈니스 도메인별 독립 모듈
- **B**: 사용자 기준 (고객 API, 관리자 API, 공통) — 사용자 유형별 분리
- **C**: 레이어 기준 (Router 모듈, Service 모듈, Repository 모듈) — 기술 레이어별 분리

[Answer]: A
